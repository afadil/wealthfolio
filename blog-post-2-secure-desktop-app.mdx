# A Technical Deep Dive into Wealthfolio's Architecture: Present and Future

Wealthfolio is a privacy-focused, open-source investment tracker that is constantly evolving. In this technical paper, we'll take a deep dive into its architecture, exploring both its current state and its exciting future, as outlined in the official roadmap. This will give you a comprehensive understanding of how Wealthfolio is built for security, reliability, and future growth.

## The Current Architecture: A Hybrid Approach

Wealthfolio is a desktop application built with a modern tech stack that prioritizes security and performance. It's a Tauri application, which means it has a Rust backend and a web-based frontend. This hybrid approach allows for a rich user experience, while leveraging the power of Rust for the core business logic.

### Frontend-Backend Communication: Tauri Commands and `serde`

The communication between the React frontend and the Rust backend is handled by Tauri commands. These are Rust functions that are exposed to the frontend and can be called from JavaScript/TypeScript.

Here's an example of how the `create_account` command is defined in `src-tauri/src/commands/account.rs`:

```rust
// src-tauri/src/commands/account.rs

#[tauri::command]
pub async fn create_account(
    account: NewAccount, // The payload from the frontend
    state: State<'_, Arc<ServiceContext>>,
    handle: tauri::AppHandle,
) -> Result<Account, String> {
    // ... implementation ...
}
```

The `NewAccount` struct, which is the payload from the frontend, is defined in `wealthfolio-core`. For this to work, the struct needs to derive `serde::Deserialize`.

```rust
// wealthfolio-core/src/accounts/accounts_model.rs
use serde::{Deserialize, Serialize};

#[derive(Queryable, Identifiable, Serialize, Deserialize, Debug, Clone, AsChangeset)]
#[diesel(table_name = crate::schema::accounts)]
pub struct Account {
    // ... fields ...
}

#[derive(Deserialize, Insertable, Debug)]
#[diesel(table_name = crate::schema::accounts)]
pub struct NewAccount {
    pub name: String,
    pub institution: String,
    // ... other fields ...
}
```

When the frontend calls the `create_account` command, it passes a JSON object that corresponds to the `NewAccount` struct. Tauri, with the help of `serde_json`, automatically deserializes the JSON object into a `NewAccount` instance. Similarly, the `Account` struct that is returned by the function derives `serde::Serialize`, so it's automatically serialized into a JSON object before being sent back to the frontend. This seamless serialization and deserialization is one of the key features that makes Tauri so powerful.

### The Market Data Sync Process

The market data sync process is a good example of the backend's capabilities. It's orchestrated by the `MarketDataService` in `src-core`, which uses a `ProviderRegistry` to manage the different market data providers.

The `ProviderRegistry` is initialized with a list of provider settings from the database. When a sync is requested, the `MarketDataService` iterates through the enabled providers and fetches the required data. This modular design makes it easy to add new providers in the future.

### The REST API Server

The centerpiece of the future architecture is a new REST API server. This server would encapsulate the `wealthfolio-core` library and expose its functionality through a set of RESTful endpoints. A good choice for building this server would be a modern Rust web framework like **`axum`**, which is known for its ergonomics, modularity, and tight integration with the `tokio` ecosystem.

### The Clients: Mobile, Web, and Desktop

With the REST API in place, Wealthfolio will be able to support multiple clients:

*   **Mobile App**: The roadmap mentions a mobile app for iOS and Android, built with Tauri Mobile. This would allow for a high degree of code reuse between the desktop and mobile apps.
*   **Web App**: A new web app would be created, which would communicate with the REST API server. This would make Wealthfolio accessible from any device with a web browser.
*   **Desktop App**: The existing desktop app would be refactored to communicate with the REST API server instead of directly with the `src-core` library.

### Challenges of the New Architecture

This new architecture presents a number of challenges that will need to be addressed:

*   **Authentication and Authorization**: A robust authentication and authorization system will be needed to secure the REST API.
*   **Data Synchronization**: A strategy for synchronizing data between the different clients will be crucial. This is a complex problem, especially in an occasionally connected environment.
*   **Deployment**: The REST API server and the web app will need to be packaged and deployed, likely as Docker images. This will require a CI/CD pipeline and a hosting solution.

## Conclusion: A Robust Foundation for Future Growth

Wealthfolio's current architecture is a solid foundation for a secure and performant desktop application. The future architecture, as outlined in the roadmap, is a bold vision that will transform Wealthfolio into a multi-platform ecosystem. While there are many challenges to overcome, the choice of Rust and its rich ecosystem of libraries provides a strong foundation for success.
